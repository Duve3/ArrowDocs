## What is GitHub?????

Git is a tool created by the legendary developer Linus Torvalds - the same guy who made Linux. It is a `version control system` meaning it helps you manage versioning within your code.

Lets explain it better by imagining something... Imagine it is the day before your district competition, you doing well makes or breaks whether you go to Worlds this year, and you change a small part of the code to be "more clean",
but now the whole robot doesn't work, it's one AM and nobody understand what the code was before. Now if you had Git set up, this would never be an issue! You would've commited the code from earlier and been able to run a simple command and rollback to it.

Now don't get it confused:
- Git is the actual verison control system
- GitHub is a website that allows for you to upload your repositories (aka your codebase) for anyone to clone and work with you on

Now throughout this tutorial we will go more in depth on Github usage, but heres a quick summary (it's okay if this doesn't make full sense yet! As you do each of the tutorials, each part will be explained in more depth!):
- A repositority is a bunch of code (each project will have its own repo)
- A branch is a different version of the code that is not tested/is WIP
- Main/Master is the branch that working, running code will always exist (Main should **ALWAYS** be good, even if it's one or two features behind)
- A commit is you signing off on your current changes and putting a message along with it (you should always commit before you leave)
- "pulling" is to pull from Github.com and to update your local code from git
- "pushing" is to push your code to github.com (you need to do this for other people to see your code)
- "merge conflict" is when two versions of the code exist (like if the version on github.com and your local changes conflict with eachother), so you need to figure out what parts to keep and what parts to remove
- All new code go onto branches which follow the naming strategy of: "feature/_" for features and "fix/_" for bugs
- A "pull request" (or PR) is what you do once your branch has been tested and works, it is requesting that your branch be merged into the main branch (aka the main robot code)
- "Code review" is the process in which peers and mentors of the team review your PR and verify that it is good, working code, before either approving it to be merged, or requesting changes before merging it.

Obviously this a non-exhaustive list of vocabulary, and these definitions aren't complete, but it's a good starter.

All of these terms will come up again as you go through each tutorial, and will be explained in much higher depth.
