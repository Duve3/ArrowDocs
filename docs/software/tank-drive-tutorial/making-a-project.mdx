# Making a Project
Lets begin our journey of tank drive by Making a Project

:::note

It's completely fine to not know what tank drive is, this tutorial will be extremely in depth (to the point of uselessness) as it is the first in the series of beginner tutorials in the documentation.

<span className="subtext">this tutorial does assume you read the [Configuring your environment](/software/configuring-your-env) and [What is Software](/software)</span>

:::

First step is to open WPILib VSCode (either called year WPILib VSCode, or just VSCode on macOS)

Then click the WPILib icon in the top right (should say "open WPILib Command Palette"). It should open a menu that looks like this:
![WPILib Command Palette](./img/WPILibCommandPalette.png)

Now we will type `>WPILib create a new project` (including the `>WPILib` that is automatically filled) and press enter.

After pressing enter you should get to a menu that looks like this:
![WPILib New Project Creation Screen](./img/new-project-screen.png)

First we will select our project type of `Template`, then for our language we will select `Java`. For our project base we will select `Command Robot`

The reasoning behind all of this is kind of confusing, but basically we are selecting from some premade java templates that will write out the basic starter code for us!

Next we will select our `Base Folder` you want to either make a new directory, or select a directory where other projects exist. 
This directory will become your 'FRC Projects' sorta directory, we want to use this one for all our FRC projects! 
:::note

of course you do not have to select a good directory, you can even just put the project on C:/, the above statement is just good practice!

:::

Next we need a project name, this can be anything, but for our project I will pick `TankBotTutorial`.

Let's also make sure `Create a new folder?` is checkmarked.
:::warning

if you do not check this box, then the project will be created in the directory you selected instead of making a new folder named after the project name.

:::

Next we put in our team number; for ArrowDynamics, this is `10079`. It's important that this is correct as messing it up can result in some very weird bugs!

Finally, we turn on `Enable Desktop Support`. This allows us to simulate the robot, while WPILib does give a warning about this with VenderDeps (okay if you don't know what that is yet!) but I personally haven't experienced any issues with it enabled.

And once you press `Generate Project` you will have officially made your first WPILib project!!

## Let's Look At What We Generated!

You should have a folder directory that looks like this:
<img src={require("./img/TankBotDirectoryUponCreation.png").default} style={{width: 20 + "rem"}}/>

This is a lot of files! So let's break down what each file/directory does!

### Simple Things
`WPILib-License.md` is just the license for using WPILib\
`.gitignore` is a file used by Git to understand what files can be ignored\
`.wpilib/wpilib_preferences.json` is the file used for storing your project's configuration\
`.vscode/` is the directory that contains a bunch of VSCode configuration\
`build/` contains the artifacts after building, you don't need to interact to this most of the time!

### Gradle
When writing code, you need some kind of compiler or build system. Most Java programs use *Gradle*.

Gradle's job is to look at the build.gradle and the settings.gradle file, then based off of those, compile your Java code!
<span className="subtext">Don't worry, you don't have to deal with Gradle most of the time! (WPILib handles most of it for you!)</span>

Gradle is a more complicated topic, but it will be explained as we go and interact with their files.
Speaking of their files!
The following items are related to Gradle:
- `gradlew` & `gradlew.bat` are the actual gradle executable
- `settings.gradle` handles some settings for WPILib, mainly looking for your WPILib install!
- `build.gradle` is the main file, it configures most of the build process, don't worry we won't touch this file for a while!
- `.gradle/` is a directory that contains everything gradle generated as it ran, you can completely ignore this directory
- `gradle/` contains gradle_wrapper, its a jar file that handles gradle itself by wrapping around it and giving a better interface to users

:::note

You will NOT have to interact with gradle that often! This is just an explaination so you understand the *why* behind each generated file!

:::

### Source Files
All code will be located in the `src/` directory. Inside you'll find `main/` (VSCode often compresses these as there's only one folder inside of src)

Inside of `main/` you'll find `deploy/` and `java/` (aka `java/frc/robot/`). We will talk about those in the next part of this tutorial!

The former (`deploy/`) contains files that will be deployed onto the robot itself along with your code.
Obviously your code is deployed onto the robot, but sometimes you need to put some other files (like configuration files) onto the robot for your code to read.

Files like that would be placed in this `deploy/` directory.
