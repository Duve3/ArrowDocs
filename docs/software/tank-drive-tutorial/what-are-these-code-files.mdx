# What do all these code files do??!?!?

Now that we have generated all these files, and understood what they all do, we can now understand one last file type.\
The files located within `src/main/java/frc/robot`. These files are the actual source code of your robot.\

Let's delve into what each file does!

:::tip

Each programmatic (.java) file has comments in it that helps explain in even more depth.

I recommend that you read through each file mentioned, as the comments that are generated by default are really good for learning!

:::

## `Constants.java`
This is the simpliest file on the list, it's designed to contain every major constant used within your robot.

The reason you have a Constants file is because it allows you one place to configure values used everywhere in the robot.

One thing to notice here is that while it is one file, the class itself is sectioned into smaller static classes:

```java showLineNumbers title="Constants.java"
// ...
public final class Constants {
    // each 'subset' of constants are implemented as their own static class with static values
    // meaning you would call kDriverController as Constants.OperatorConstants.kDriverControllerPort
            // highlight-start
    public static class OperatorConstants {
        public static final int kDriverControllerPort = 0; 
    }
            // highlight-end
}

// ...
```

## `RobotContainer.java`
This is where basically everything comes together on your robot, it's where you will probably end up looking at the most, as the way it is setup defines the whole robot.

This file is quite complicated, so lets go over it in parts!

The first part of the class is definitions, when writing your class you should always define your controllers, subsystems, and whatever else here.

The next part is known as the constructor, its the part that actually creates the class, as you can see the constructor takes in no parameters and just runs one function `configureBindings();`
Constructors usually just contain all your initalization steps; in this case, that is just `configureBindings();`

Now configure bindings is a little more interesting, it's where you define how your controller interacts with your subsystems, or any other triggers that occur upon certain conditions.

`configureBindings();` is just saying "do X when Y is Z" for example, "run our shooter WHILE the driver presses the button B"

The final part is `getAutonomousCommand()` it's used to define what your Auto command is!

For now we won't go more in depth on Autos, as it is a more complicated topic that we will address later, for now just understand that your robot pulls from here for autos!

:::note

Don't know what an auto is? I recommend you watch the [2026 Game Animation](https://www.youtube.com/watch?v=_fybREErgyM)!

:::

```java showLineNumbers title="RobotContainer.java"
// ...

public class RobotContainer {
    // our definitions!
    private final ExampleSubsystem m_exampleSubsystem = new ExampleSubsystem();

    // hey look! We are using our OperatorConstants here!
    private final CommandXboxController m_driverController = 
        new CommandXboxController(OperatorConstants.kDriverControllerPort);

    // Our constructor!
    public RobotContainer() {
        configureBindings();
    }

    private void configureBindings() {
        // We are saying that once a certain condition is met, run this command!
        new Trigger(m_exampleSubsystem::exampleCondition)
            .onTrue(new ExampleCommand(m_exampleSubsystem));

        // While the driverController is pressing B, run this example command
        m_driverController.b().whileTrue(m_exampleSubsystem.exampleMethodCommand());
    }

    public Command getAutonomousCommand() {
        // Your robot will run this example command during Auto
        return Autos.exampleAuto(m_exampleSubsystem);
    }
}
```

## `Robot.java`
This file is a little different, you'd probably think that this file would have all your robot code, but because of [Encapsulation, and a few other reasons](https://docs.wpilib.org/en/stable/docs/software/commandbased/structuring-command-based-project.html#robot)
you'll find that this file is actually quite empty!

This is the class that actually extends the fundamental [`TimedRobot`](https://github.wpilib.org/allwpilib/docs/release/java/edu/wpi/first/wpilibj/TimedRobot.html) that handles everything by WPILib.

Similar to `RobotContainer.java`, we define just a few things at the top of the file; our `robotContainer`, and our autonomousCommand!

Next we have our constructor, so we define our `RobotContainer`

After that you'll notice a lot of functions that all override from the `TimedRobot`:
- `disabledInit()`
- `disabledPeriodic()`
- `autonomousInit()`
- `autonomousPeriodic()`
- `teleopInit()`
- `teleopPeriodic()`
- `testInit()`
- `testPeriodic()`
- `simulationInit()`
- `simulationPeriodic()`

Whats important is that all functions ending in `Init` are only ran just on the beginning of entering a certain phase (auto, teleop, disabled, etc.)\
All other functions ending in `Periodic` are ran every **20ms** during a certain phase.

Let's go over what each phase is:
- `disabled` this phase is when the robot is unable to move or do any actions (motors will NOT move during this phase!)
- `autonomous` this phase is that 15-30 second phase at the beginning of a match where the robot controls itself
- `teleop` this phase is that phase where humans control the robot (the rest of the match)
- `test` this phase is when testing the robot, such as code that automatically runs a system test for more info look here: [https://docs.wpilib.org/en/stable/docs/software/basic-programming/using-test-mode.html](https://docs.wpilib.org/en/stable/docs/software/basic-programming/using-test-mode.html) 
- `simulation` this is always ran every 20ms while simulating the robot (simulation is a higher level topic that will not be covered until later!)

On some specifics, `robotPeriodic()` just runs the commandScheduler (it's always called every 20ms, no matter what)
`autonomousInit()` pulls the autonomous command from `RobotContainer` and schedules it to run!

:::tip

Still don't get what exactly the phases are, and what they do? Consider watching the [2026 Game Animation](https://www.youtube.com/watch?v=_fybREErgyM)!

:::

```java title="Robot.java"
// ...

public class Robot extends TimedRobot {
    // Our definitions!
    private Command m_autonomousCommand;

    private final RobotContainer m_robotContainer;

    public Robot() {
        // lets initalize our robotContainer!
        m_robotContainer = new RobotContainer();
    }

    @Override
    public void robotPeriodic() {
        // running any active commands during robotPeriodic!
        CommandScheduler.getInstance().run();
    }

    // by default, we don't want to do anything while disabled!
    @Override
    public void disabledInit() {}

    @Override
    public void disabledPeriodic() {}

    // when our auto begins, lets run our autoCommand!
    @Override
    public void autonomousInit() {
        m_autonomousCommand = m_robotContainer.getAutonomousCommand();

        // if we have an autonomousCommand, lets schedule it
        if (m_autonomousCommand != null) {
            CommandScheduler.getInstance().schedule(m_autonomousCommand);
        }
    }

    @Override
    public void autonomousPeriodic() {}

    @Override
    public void teleopInit() {
        // to ensure human control of autonomous command, we immediately cancel it once teleop begins
        if (m_autonomousCommand != null) {
            m_autonomousCommand.cancel();
        }
    }

    @Override
    public void testInit() {
        // cancels all the commands when entering test mode
        CommandScheduler.getInstance().cancelAll();
    }

    // for the rest of these we don't really need to do anything
    @Override
    public void testPeriodic() {}

    @Override
    public void simulationInit() {}

    @Override
    public void simulationPeriodic() {}
}
```

## `Main.java`

This file is the most simple file. Like any other Java program, it is the entrypoint of the program!

To practice good encaspulation (and the other reasons mentioned up above for `Robot.java`), this file is extremely small; all it does is run RobotBase.startRobot(Robot::new);

Let's look a little closer in this actual code
```java title="Main.java"
public final class Main {
    // this class cannot be initalized
    private Main() {} 

    public static void main(String... args) {
        // this is a little confusing but the way it works is its calling the WPILib function that initalizes and handles robot creation (RobotBase.startRobot) 
        // and then calls our Robot.java's Robot::new function 
        RobotBase.startRobot(Robot::new);
    }
}
```
